# IntensiCare v2 — Clinical Hazard Log (ISO 14971)

**Owner:** clinical-safety-officer (patient-safety veto holder) · **Version:** 1 · **Verification pass:** Phase C (2026-07-04) · **Machine source of truth:** `docs/plan/_work/safety/hazard-log.yaml`

This log follows the ISO 14971 chain — *hazard → hazardous situation → harm* — with initial risk judged as severity × probability before controls. It was seeded from the legacy audit's verified P0 escalations (12), the cross-cutting systemic defect classes SYS-01..10 (10), and the inherent hazards of any real-time clinical-decision-support platform (12). Every mitigation is a **testable design requirement with a named owner**; a mitigation is considered implemented in the *design* only when it is traced to a concrete section of the v2 specification (`spec_ref` — domain specs under `clinical/domains/`, engine and data specs under `architecture/`, screen specs under `design/screens/`, the canonical units registry, the severity model, and the latency/PPV budgets). 93 of 105 mitigations are now spec-traced; the remainder are marked *(open)*. `residual_risk` is set only where the load-bearing mitigations were verified in those specs; **status `mitigated`** means every mitigation is traced, while **status `open`** means a spec anchor, a clinical-committee RATIFY sign-off (mandatory for every P0-band rule per SYS-C-03 — the corrected behavior is designed to a recommended default but a human clinical decision still gates firing), or a named test verification is still pending. Nothing in this log authorizes porting legacy defect behavior: suppression may never mask a severity upgrade, critical delivery is never rate-limited, and every unit crossing the computation boundary must be canonical.

**Risk vocabulary.** Severity: catastrophic / critical / serious / minor. Probability and risk: high / medium / low. Counts below: 5 catastrophic, 20 critical, 9 serious, 0 minor.
