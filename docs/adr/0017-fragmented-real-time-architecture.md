# 0017. Fragmented real-time architecture across polling, WebSocket, and Firestore

Status: superseded by ADR-0019
Date: 2026-07-03
Audit source: trilhas-frontend @ f9656be2660ec2048ce6240b4ac418b7fe7d5a5b

## Context and Problem Statement

The legacy platform needs live updates in at least three places that all ultimately react to the same underlying event stream (a bed's clinical alert changing severity): the notification bell, sector chat/presence, and the ICU bed board/dashboard. Rather than one push transport serving all three, the legacy codebase grew three independent, unrelated real-time mechanisms, each added when a specific feature needed it, with no shared abstraction for connection lifecycle, reconnection, or backoff (inventory §6.2). Firebase is a project dependency, yet the single most safety-critical view — the bed board — does not use it, or any push channel, at all (inventory Summary #5). Any rebuild has to decide whether to keep three transports or standardize on one, and how to preserve the sound part of the legacy design (Firestore never became a system of record) while fixing the part that is not sound (the bed board lagging the alert that drives the bell).

## The Legacy Decision

- **Notifications (raw bidirectional WebSocket, two connections):** `DisplayNotificaoes` opens two separate `w3cwebsocket` connections to `${baseWsURL}notificacao/?token=…` — one for the notification list, one with `&popover=true` solely to trigger a `notification.open()` toast (debounced 2s via `trotthleNotification`). Marking a notification read sends `{evento:"visualizar", visto:true, id}` back over the *same* socket, i.e. the socket doubles as both a read channel and a mutation channel. Neither connection has reconnect or backoff logic; `ws.onerror` only logs to console. `trilhas-frontend:src/components/DisplayNotificaoes/DisplayNotificaoes.tsx:1-133`
- **Chat/presence (Firestore `onSnapshot`, narrow signaling layer only):** every Firestore access in the app targets exactly one document shape, `chats/{setorId}/usuarios/{usuarioId}`. `useCollection`/`useDocument` subscribe via `onSnapshot` and re-run a callback on every emission. `trilhas-frontend:src/hooks/Firestore/useCollection.tsx:1-77`. `MessagesList` uses that single document only as a trigger: when it changes, it re-runs the REST message fetch (`_getMensagensBySetor`) and, if the panel is active, zeroes the unread counter (`qtd_mensagens: 0`) back on the same document — chat history itself is always paginated REST, never read out of Firestore. `trilhas-frontend:src/components/MessagesList/MessagesList.tsx:332-348`. The same document also carries `online_call: boolean` for video-room presence elsewhere in the app (inventory §6.2/§6.3).
- **Bed board / dashboard (REST polling via `setInterval`, per-tenant configurable):** `ListOcupacoes` polls with `setInterval(() => applyFilters({...filters}), reloadTime * 1000)`, started/stopped by an app-wide `AutoReloadContext.update` boolean toggled from a header `Switch`. `trilhas-frontend:src/contexts/AutoReloadContext/AutoReloadContext.tsx:1-22`; `trilhas-frontend:src/components/ListOcupacoes/ListOcupacoes.tsx:112-141`. `reloadTime` is populated from the per-tenant business setting `empresa.tempo_atualizacao` (confirmed via `reloadTime` prop threading back to the `FormEmpresa`/`empresa` record; inventory §6.2). No WebSocket or Firestore subscription exists anywhere in the bed-board or dashboard code paths.
- Net effect: the same underlying alert event reaches the bell in real time (WebSocket push) but reaches the grid only on the next poll tick, up to `tempo_atualizacao` seconds later — the two live views of one event can visibly disagree for that window (inventory Summary #5, §6.2).

## Evident Rationale

*(Inferred — not stated in the codebase.)* Each channel was plausibly bolted on independently, at the moment a specific feature needed "live-ish" behavior, using whatever was already closest at hand: notifications got a Django-Channels-style raw WebSocket because push-to-a-bell is the classic WebSocket use case and no shared client existed yet to reuse; chat/presence reused Firestore because Firebase was already a project dependency for something else (most likely originally adopted for `agora-rtc-react` video presence) and `onSnapshot` is the path of least resistance for "tell me when something changed," while the team correctly kept REST as the durable store rather than trusting a NoSQL side-channel with clinical/chat history; the bed board got the simplest possible mechanism, periodic refetch, likely because it predates the WebSocket work or because polling is trivially safe and framework-agnostic. Making the interval a per-tenant setting (`empresa.tempo_atualizacao`) suggests real-world tuning against variable network quality and device capability across ICU deployments — a legitimate operational need, just solved at the wrong layer (interval tuning instead of push).

## Assessment

**Strengths:**
- The Firestore boundary is a genuinely sound principle, held consistently: exactly one narrow document shape, used only for signaling/presence, with REST remaining the system of record for both chat history and clinical data. `trilhas-frontend:src/hooks/Firestore/useCollection.tsx:1-77`; `trilhas-frontend:src/components/MessagesList/MessagesList.tsx:332-348`
- Each mechanism, taken alone, is simple and legible — a new engineer can understand what any one of the three does in isolation without needing to learn a shared real-time framework.

**Weaknesses:**
- **No shared reconnect/backoff anywhere.** The WebSocket client has none (`ws.onerror` just logs), so a dropped connection silently stops delivering notifications until the user reloads the page. `trilhas-frontend:src/components/DisplayNotificaoes/DisplayNotificaoes.tsx:1-133`
- **The most safety-critical view is the least live.** The bed board — the primary surface a clinician watches for deteriorating patients — is REST-polled on a per-tenant interval, while the notification bell for the *same* alert is pushed instantly over WebSocket. A red (VERMELHO) alert can appear in the bell immediately and take up to `tempo_atualizacao` seconds to reach the grid. `trilhas-frontend:src/contexts/AutoReloadContext/AutoReloadContext.tsx:1-22`; `trilhas-frontend:src/components/ListOcupacoes/ListOcupacoes.tsx:112-141`
- **A raw bidirectional socket used as a mutation channel** (`clearMessages` sends `visualizar` events back over the same read socket) couples read and write semantics onto one connection with no documented protocol/versioning, making the wire contract implicit and hard to evolve safely. `trilhas-frontend:src/components/DisplayNotificaoes/DisplayNotificaoes.tsx:1-133`
- **Two live WebSocket connections per user for one feature** (list + popover) double the connection/auth overhead for logically one notification stream, rather than multiplexing one connection. `trilhas-frontend:src/components/DisplayNotificaoes/DisplayNotificaoes.tsx:1-133`
- **Polling is gated by a manual, app-wide toggle** (`AutoReloadContext.update`, a header `Switch`) rather than being an always-on guarantee — a clinician can inadvertently leave auto-refresh off and not know the board has stopped updating at all, with no visible staleness indicator. `trilhas-frontend:src/contexts/AutoReloadContext/AutoReloadContext.tsx:1-22`
- **Three unrelated client libraries/protocols** (raw `websocket` package, Firebase 8 Firestore SDK, `setInterval`) for what is conceptually one problem (push a server-side event to a subscribed client) means three sets of connection-lifecycle edge cases (tab backgrounding, token expiry mid-connection, network flaps) to reason about separately, none of them handled today.

## Considered Options

1. **Lift and shift**: keep all three mechanisms as-is. Rejected outright — it carries forward the life-safety risk (bell/grid disagreement) and the missing-reconnect defect verbatim into a new platform; there is no part of this pattern worth preserving as a *mechanism*, only the Firestore-boundary *principle*.
2. **Single WebSocket (or SSE) channel with a shared client, topic-based subscriptions, and shared reconnect/backoff**, feeding notifications, chat-refresh signaling, presence, and bed-board/dashboard updates as distinct message topics over one connection per client. REST remains the system of record for all durable data (clinical records, chat history, audit trail); the realtime channel only ever carries "something changed, refetch/patch" signals or thin payloads, never becomes authoritative.
3. **Managed realtime subscription service** (e.g., a hosted pub/sub or realtime-database product, potentially still Firestore/Firebase but used more broadly than today) as the one shared transport, with the same system-of-record discipline as Option 2 — trades operational ownership of a WebSocket/SSE server for a vendor dependency and its cost/latency/vendor-lock-in profile.
4. **Keep polling for the bed board, but shorten and fix the interval / remove the manual off-switch.** Rejected as insufficient on its own — even a short poll still cannot match WebSocket-push latency for the bell, so the fundamental bell/grid disagreement persists, just at a smaller magnitude; it does not address the missing-reconnect defect on the notification socket either.

## Decision Outcome

Recommend **Option 2**: standardize the new platform on a single push transport (WebSocket or SSE, chosen once the backend framework/hosting is decided) with one shared client implementing reconnect-with-backoff, connection-state visibility, and topic/channel-based multiplexing, so that notifications, chat-refresh/presence signals, and bed-board/dashboard updates all ride the same guaranteed-consistent pipe — eliminating the possibility that the bell and the grid disagree about the same event. Explicitly preserve the one sound principle from the legacy design: the realtime layer is a signal/transport only, never a second source of truth — clinical records, chat history, and audit data continue to live in the durable datastore (REST/DB-backed API), and any realtime payload either triggers a refetch or carries a thin, idempotent patch that the client reconciles against the authoritative record. Option 3 (a managed pub/sub/realtime product) is an acceptable substitute for the *transport* if the team prefers not to operate WebSocket/SSE infrastructure directly, provided it is used broadly (all three use cases) rather than narrowly, and provided the same system-of-record discipline is enforced. This is a recommendation pending team ratification; the specific transport (self-hosted WebSocket/SSE vs. a managed realtime product) should be settled alongside the backend framework and hosting decisions for the new platform.

### Consequences

**Good:**
- Removes the life-safety risk at the center of this ADR: the bed board and the notification bell become guaranteed to reflect the same event at the same time, since both subscribe to the same push channel instead of two channels with different latency characteristics.
- One shared reconnect/backoff implementation replaces zero implementations today, closing the silent-disconnect defect in the current notification socket.
- Reduces per-client connection overhead (one multiplexed channel instead of two WebSocket connections plus a Firestore subscription plus a poll timer) and gives one place to add connection-state UI (e.g., a "reconnecting…" indicator), which the legacy platform has nowhere today.
- Retiring `setInterval`-based polling for clinical data removes the per-tenant `tempo_atualizacao` tuning knob as a life-safety-relevant setting — staleness stops being something an administrator can inadvertently mis-configure.

**Bad:**
- Consolidating three independent mechanisms into one shared transport is a larger upfront engineering investment than continuing to bolt on point solutions, and it introduces a new piece of shared infrastructure (a WebSocket/SSE gateway or managed subscription service) that must itself be made highly available, since it now serves a life-safety-relevant view.
- A single shared channel means an outage or bug in the realtime layer now affects all three use cases at once (notifications, chat/presence, bed board) rather than being isolated per feature as it is today — the blast radius of a realtime-layer defect grows, which raises the bar for testing and monitoring that channel.
- Teams must actively resist the temptation to let the realtime channel become a de facto cache or source of truth as more features are layered onto it over time; this requires ongoing discipline (and likely automated checks) rather than being structurally prevented.
