## Purpose

This report lists **every rule that requires a human clinical or business decision before it is reimplemented.** It is the escalation queue for the legacy IntensiCare (trilhas) rule audit.

**Nothing in this document has been corrected.** Every rule here is captured in the reconciled catalog **exactly as the legacy system behaves** — deviations, unit mismatches, dead branches, off-by-one boundaries and inverted predicates are all preserved verbatim. The audit's job was to *find and describe* these items faithfully, not to fix them. A clinician or the product owner must decide, per item, whether the legacy behavior is intended (and should be carried forward) or is a defect (and should be changed) — that decision is explicitly **out of scope** for the audit and is why each item is escalated.

Items are ranked by clinical impact (high → moderate → low → none → n/a) and, within a band, clinical categories (clinical-scoring, drug-dosing, physiological-calculation, alert-threshold, care-pathway, triage-eligibility) are listed before administrative ones. **Each rule appears in exactly one section** (its highest applicable band); it is never duplicated.

## Cross-cutting systemic issues

The escalations are not independent. The same handful of root causes recur across clusters. Each pattern below lists the affected rule IDs so a reviewer can fix (or ratify) the class once rather than case by case. Rule IDs may repeat across patterns because a single rule can embody more than one defect class.

### 1. FiO2 percent-vs-fraction inconsistency

`DadosProntuario.FiO2Validator` constrains FiO2 to `0` or `21..100` (a **percentage**), yet the P/F ratio and every threshold that consumes it (SOFA respiratory 400/300/200/100; ventilation weaning 150–300; ARDS/Berlin) assume FiO2 as a **fraction 0.21–1.0**. The codebase is internally inconsistent: some sites divide FiO2 by 100, some compare it as a fraction directly, and the unit tests set `fio2=1` attribute-directly, bypassing the validator and masking the defect. With validator-conformant (percentage) data the ratio is ~100× too small, driving the SOFA respiratory sub-score to the maximum and spuriously classifying severe ARDS, while fraction-only comparisons (e.g. `fio2 < 0.4`) never fire on percentage data.

Affected: RULE-CLINICAL-SCORING-002, RULE-CLINICAL-SCORING-008, RULE-VENTILACAO-004, RULE-VENTILACAO-005, RULE-VENTILACAO-006, RULE-VENTILACAO-011, RULE-SEDACAO-004, RULE-SEDACAO-016, RULE-SEPSE-020, RULE-EVOLUCOES-001, RULE-TRILHAS-ENGINE-017.

### 2. Vasopressor dose-unit chaos (ml / ml/h vs mcg/kg/min vs mcg/kg/h)

Noradrenaline and other vasopressor/inotrope thresholds are expressed in incompatible units across the codebase. Some predicates read a **raw infusion volume** (`Noradrenalina.quantidade` in ml) or an **infusion rate in ml/h**; the clinical standard (and the facade text a clinician reads) is a **weight-based rate in mcg/kg/min**. ml/h → mcg/kg/min is not convertible without drug concentration and patient weight, so the trigger and the displayed regimen do not correspond. A separate min-vs-hour inconsistency exists between sibling pathways: the same `>0.5` noradrenaline cutoff is labelled **mcg/kg/min** in the estabilidade facade (RULE-ESTABILIDADE-016) but **mcg/kg/h** in the estabilizacao sibling (RULE-ESTABILIDADE-024) — a 60× discrepancy on the same number.

Infusion-volume-vs-rate: RULE-CLINICAL-SCORING-005, RULE-ESTABILIDADE-007, RULE-ESTABILIDADE-008, RULE-ESTABILIDADE-009, RULE-ESTABILIDADE-016, RULE-ESTABILIDADE-019, RULE-SEDACAO-005, RULE-SEDACAO-010, RULE-SINAIS-VITAIS-029, RULE-SEPSE-014.
Per-minute-vs-per-hour label drift: RULE-ESTABILIDADE-007, RULE-ESTABILIDADE-008, RULE-ESTABILIDADE-016, RULE-ESTABILIDADE-019, RULE-ESTABILIDADE-024, RULE-SEDACAO-010, RULE-SINAIS-VITAIS-029, RULE-TRILHAS-ENGINE-017.

### 3. Lactate units mg/dL vs mmol/L

Lactate thresholds are used without a consistent unit convention; the sepsis / shock pathways and their facades disagree on whether a lactate value is mg/dL or mmol/L (roughly a 9× conversion factor), so the same numeric cutoff denotes very different severities depending on the ingested unit.

Affected: RULE-EFICIENCIA-008, RULE-ESTABILIDADE-005, RULE-ESTABILIDADE-020, RULE-SEPSE-013, RULE-SEPSE-050, RULE-SEPSE-058, RULE-SEPSE-059, RULE-SEPSE-061, RULE-VENTILACAO-012.

### 4. Facade-text vs model-predicate threshold drift

The clinician-facing recommendation/alert text (the `facade` layer: `get_detalhe`, `get_payload_*`, `payload_*_v3`) states a threshold, unit, or numbering that the actual v3 model predicate never evaluates. A clinician reads a mcg/kg/min dose, an FC > 130 bpm criterion, or a criterion number that the trigger did not verify — the displayed rationale and the firing condition diverge.

Affected: RULE-ALERTAS-003, RULE-ESTABILIDADE-007, RULE-ESTABILIDADE-009, RULE-ESTABILIDADE-011, RULE-ESTABILIDADE-012, RULE-ESTABILIDADE-015, RULE-ESTABILIDADE-016, RULE-ESTABILIDADE-024, RULE-EFICIENCIA-002, RULE-EFICIENCIA-003, RULE-PIORA-CLINICA-002, RULE-PIORA-CLINICA-003, RULE-PIORA-CLINICA-011, RULE-PROFILAXIA-008, RULE-SEDACAO-020, RULE-SEPSE-043, RULE-SEPSE-058, RULE-SEPSE-059.

### 5. v1-vs-v3 AND/OR aggregation divergence (absence-filter AND-collapse)

Criteria that the reference (and the docstring) define as **OR alternatives** are implemented as a single Django `.filter()` with comma-separated (**AND**) conditions, or vice-versa. The most dangerous instance is the "no sedation / no vasopressor" **confounder-exclusion** clauses: an AND across all sedatives requires every drug to be present simultaneously (practically never true), so the exclusion never fires and a fully sedated patient can be flagged. The mirror defect drops OR-alternative positive criteria so an alert under-fires.

Affected: RULE-EFICIENCIA-005, RULE-EFICIENCIA-008, RULE-EFICIENCIA-010, RULE-SEDACAO-004, RULE-SEDACAO-008, RULE-SEPSE-014, RULE-SEPSE-021, RULE-PIORA-CLINICA-010.

### 6. Chained-comparison Python misparse suppressing the severe band

Expressions of the form `if 7 <= dor > 10:` are parsed by Python as `(7 <= dor) and (dor > 10)`. Because the input validator caps the scale at its maximum (10, 12, …), the second clause is impossible, so the **most severe band never fires** — the highest-acuity pain/agitation grade is silently suppressed. Intended code was `low <= x <= high`.

Affected: RULE-PIORA-CLINICA-006, RULE-PIORA-CLINICA-007, RULE-SEDACAO-018.

### 7. Off-by-one / exclusive-range boundary (dead gaps at band edges)

Two flavors: (a) Python `range(a, b)` stop bound is exclusive, so boundary values (e.g. status 399, creatinine 5.0) fall through to the wrong branch; (b) strict `<` / `>` used where the reference uses `<=` / `>=`, leaving uncovered gaps at band edges that resolve to `None`/0 — e.g. SOFA creatinine `(4.9, 5.0]`, bilirubin `[1.9,2.0)`, `[5.9,6.0)`, `[11.9,12.0)`. These undercount at the top of the scale, exactly where acuity is highest.

Affected: RULE-CLINICAL-SCORING-002, RULE-CLINICAL-SCORING-004, RULE-CLINICAL-SCORING-007, RULE-AUDITORIA-LOGS-009, RULE-AUDITORIA-LOGS-010, RULE-AUDITORIA-LOGS-013, RULE-EFICIENCIA-003, RULE-EFICIENCIA-010, RULE-ESTABILIDADE-005, RULE-ESTABILIDADE-019, RULE-OPERACIONAL-INFRA-005, RULE-OPERACIONAL-INFRA-032, RULE-SEDACAO-010, RULE-SEPSE-040, RULE-SEPSE-058, RULE-VENTILACAO-012.

### 8. Dead / unwired / unreachable criterion code

Whole criteria are computed but never wired into the aggregation (`calcular_criterio_N` commented out), or gated behind a predicate that is structurally always-False (e.g. a diagnosis filter built with `vars(...).fromkeys(...)` that keys on field *names*), or written so a band is unreachable. These are silent false-negatives: the criterion appears to exist but can never contribute to an alert.

Affected: RULE-EFICIENCIA-005, RULE-SEPSE-014, RULE-SEPSE-006, RULE-SEPSE-009, RULE-SEPSE-019, RULE-CLINICAL-SCORING-007, RULE-PIORA-CLINICA-006, RULE-PIORA-CLINICA-007, RULE-PIORA-CLINICA-010, RULE-ESTABILIDADE-001, RULE-ESTABILIDADE-005, RULE-ESTABILIDADE-008, RULE-ESTABILIDADE-009, RULE-ESTABILIDADE-011, RULE-SEDACAO-003, RULE-SEDACAO-004, RULE-SEDACAO-008, RULE-SEDACAO-010, RULE-SEDACAO-011, RULE-AUDITORIA-LOGS-009, RULE-NUTRICAO-004, RULE-MOVIMENTACAO-ADT-036.

### 9. Weight decimal-separator parse (~10× inflation)

Weight is parsed with `float(''.join(re.findall(r'\d+', peso.replace(',', '.'))))`, which strips the decimal point entirely: `'70,5'` / `'70.5'` → `705` kg. Every weight-normalized threshold (mL/kg/h urine output, mcg/kg dosing) is then ~10× off, and any bed/occupancy or capacity count built on the same parsing style is likewise corrupted.

Affected: RULE-CLINICAL-SCORING-002, RULE-SEPSE-014, RULE-TENANCY-ORGANIZACAO-003, RULE-TENANCY-ORGANIZACAO-004, RULE-TENANCY-ORGANIZACAO-012.

### 10. Month-agnostic day-of-month date filtering

A shared window-construction helper filters on `criado_em__day`, which matches the **day number regardless of month/year**, so readings from the same day-number in a different month contaminate the 07:00–07:00 balance/vitals windows (over- or under-counting fluid balance, urine output, and peak temperature). A related boundary defect uses `timezone.now()` without `.astimezone()`, computing month/year boundaries in the wrong timezone and silently excluding month-end rows despite an "inclusive" intent.

Affected: RULE-BALANCO-HIDRICO-006, RULE-BALANCO-HIDRICO-007, RULE-BALANCO-HIDRICO-008, RULE-BALANCO-HIDRICO-009, RULE-BALANCO-HIDRICO-010, RULE-BALANCO-HIDRICO-011, RULE-AUDITORIA-LOGS-001, RULE-OPERACIONAL-INFRA-010, RULE-TENANCY-ORGANIZACAO-015.
